# Design: artifact lineage, task dispatch, and two deployment bindings

*Status: agreed design, pre-implementation. Companion to
[ROADMAP.md](ROADMAP.md) (roadmap item #2) and [RATIONALE.md](RATIONALE.md)
(the contract-first *why*). This file is the **design of record** for the
ground-truth/artifact model (#2), the `JobRunner` seam lit up for real, and the
two deployment bindings (Electron laptop + docker-compose server) that together
*prove* the "code once, deploy in multiple scenarios" claim.*

Last updated: 2026-05-31.

---

## 0. Why these three things are one design, not three

The request that produced this doc was: "define a handover endpoint — an
Electron app that launches the backend and *exercises task dispatch/tracking*,
so that artifacts generated by e.g. refining a prevailing model
(`giant.quick_refine`) land consistently with our emerging artifact-tracking
system."

That is a single spine, not three features:

1. **#2 artifact/ground-truth model** says: the DB owns mutable state including a
   *pointer to the current-best artifact*; bytes are write-once; imported
   JSON/CSV/YAML stays frozen.
2. **Task dispatch** (`giant.quick_refine`) is the first thing that *produces a
   new artifact*. It is the forcing function that proves #2 — if a job's output
   can only enter the system *through* the #2 model, drift is impossible **by
   construction**, not by discipline. (The prototype drifted precisely because a
   built model went to disk + Redux but never to the canonical record.)
3. **The Electron app** is just the *laptop binding* of the contract
   (`LocalFileStore` + `LocalProcessRunner`), packaging the whole loop so a
   colleague can run inspect → build → refine → land-artifact end to end.

And one addition that turns an *assertion* into *evidence*:

4. **A docker-compose binding** of the **same** `inspect_api/`. One deployment
   proves nothing — you can always make a single target work by leaking
   assumptions into the core. The proof is the *diff between two deployments*:
   if Electron-laptop and docker-compose share the exact same backend and differ
   only in (a) which `DataStore`/`JobRunner` is wired and (b)
   packaging/orchestration, the seams are real and the contract is
   backend-neutral. Two bindings is the minimum that makes the claim
   falsifiable.

## 1. Artifact lineage (roadmap #2)

### 1.1 The problem with today's schema

`Artifact` is currently a flat reference table — `(kind, relpath)` resolved
against `source_root`. It has **no lineage, no notion of "current", and no
producer**. Every artifact is implicitly an *import* artifact discovered by an
ingest reader walking the filesystem. That is exactly the shape that drifted in
the prototype: when a human built a model, the bytes hit disk and Redux, but no
row ever recorded "*this* pdb is now the best model for *this* event, and it
came from *that* action."

So #2 really asks: **what is the unit of "current-best model", and how does a
new one get created without lying about where it came from?**

### 1.2 Decision: lineage on `Artifact` (self-FK), pointers split Event/Dataset

We add lineage *to `Artifact` itself* rather than introducing a separate
`ModelVersion` table. For a **thin reference whose job is legibility**, a
self-referential `parent` FK plus three fields gives the complete lineage DAG
with no new table. (A separate version table with `is_current`/audit/locking is
what a production multi-user service would add later — the seam is already in
the right place for that hardening.)

```
Artifact  (+ new fields)
  origin:       imported | built | refined      # how these bytes came to exist
  parent:       FK(Artifact, null)              # what they were derived from
  produced_by:  FK(Job, null)                   # the action that made them
                                                #   (null for imports & interactive builds)
  created_at:   datetime

Event
  current_model: FK(Artifact, null)   # best model for THIS event's density
                                      #   (a built-ligand model)

Dataset
  current_model: FK(Artifact, null)   # best model for the WHOLE crystal
                                      #   (a refined pdb)
```

**Why two pointers, split Event vs Dataset.** This mirrors the science exactly,
so the granularity "mismatch" is a feature, not a wart:

- A human **builds a ligand into one event's local density** → an *event-scoped*
  model. `Event.current_model`.
- You **refine the whole asymmetric unit against one dataset's reflections** (a
  single pdb that may contain ligands from several events) →
  a *dataset-scoped* model. `Dataset.current_model`.

`giant.quick_refine` therefore *consumes* the dataset input pdb + its built
ligands and *emits* one refined pdb that repoints `Dataset.current_model`.

**The write-once invariant becomes mechanical.** A `built`/`refined` artifact is
always a *new row* with new bytes at a new relpath. The old row is never
mutated; the pointer just moves. Full history = walk `parent`. "What's best
right now" = one FK hop. This is the single property the prototype lacked.

```
imported pdb  ◄── built pdb (origin=built, parent=imported)
                     ◄── refined pdb (origin=refined, parent=built, produced_by=Job)
```

### 1.3 Decision: re-ingest is additive, import-scoped, and *surfaces* divergence

This is the "real design core" the roadmap flagged. A re-ingest (or PanDDA
rerun) walks the filesystem again. Policy:

```
re-ingest:
  imported artifacts (origin=imported)  -> replaced wholesale
  machine metrics (score, interesting, resolution, R-factors) -> updated in place
  decision / comment / inspected_by     -> UNTOUCHED
  built / refined artifacts (origin != imported) -> UNTOUCHED
  current_model pointer:
     - if it points at an imported artifact   -> repoint to the new import
     - if it points at a built/refined artifact -> LEAVE IT, set a flag
       (Event/Dataset.inputs_changed = True) when the underlying import bytes changed
```

**Why "surface, don't resolve" is the only defensible default.** The human's
built model *is* the scientific assertion. Silently repointing away from it on a
rerun is the inverse of the drift bug — instead of losing the import, you lose
the human work. Silently *merging* is worse: you cannot automatically reconcile
a rebuilt ligand with a re-analysed event; that is a scientific judgement. So
the honest behaviour is: keep the human artifact, keep pointing at it, and raise
a flag — "the analysis under this built model changed; a human should look."
That flag is itself a small mutable bit of state in the DB — exactly the kind of
thing #2 says the DB is *for*. The reference stays honest about what it can and
cannot automate, which fits the project's posture.

## 2. Task dispatch / tracking — the `JobRunner` seam, for real

### 2.1 The `Job` row is the hinge

`produced_by -> Job` is what makes a job's output *tracked* and *attributable*.
A `Job` row is the only door through which job-produced bytes enter the system:

```
Job
  tool:           "giant.quick_refine" | "pandda.analyse" | ...
  dataset:        FK(Dataset, null)   # refinement is dataset-scoped
  event:          FK(Event, null)     # (some jobs are event-scoped)
  spec:           JSON                 # the JobSpec — WHAT to compute, no paths
  status:         queued | running | succeeded | failed   # mirrors JobRunner.status
  runner_handle:  str                  # opaque id from JobRunner.submit()
  output_artifact: FK(Artifact, null)  # set on success
  log_relpath:    str                  # where the runner wrote stdout/stderr
  created_at, finished_at
```

The loop:

```
submit refine
  -> Job(status=queued) + JobRunner.submit(spec) -> runner_handle
  -> poll JobRunner.status(handle) -> update Job.status
  -> on success: register output bytes as
       Artifact(origin=refined, parent=<input>, produced_by=<job>)
       and repoint Dataset.current_model
```

**Every byte a job produces enters through this one door** — that is what makes
"dispatch consistent with the artifact-tracking system" true by construction.

### 2.2 The ligand-build (#4) uses the *same* artifact contract, a different producer

The "add current ligand at current location" feature (#4) is *interactive*
(client-side Coot calls), not *dispatched*. So it does **not** create a `Job`;
it registers its output directly as
`Artifact(origin=built, parent=<dataset input>, produced_by=null)` and repoints
`Event.current_model`. Same artifact shape, same write-once invariant, different
producer. Proven Coot recipe to port (from the prototype, on Moorhen 0.23 — see
`~/Developer/MoorhenPanddaApp/.../OutputFilesDatasetFilesCard.tsx:111`):

```
cootCommand get_monomer_and_position_at ["LIG", molNo, ...origin.map(-)]
  -> theMolecule.fitLigand(activeMap.molNo, ligandMol.molNo, true,true,true,10)
  -> merge_molecules -> redraw -> register Artifact + PATCH decision=hit
```

Building a ligand *is* the hit assertion → fire the decision PATCH
automatically.

### 2.3 Decision: `LocalProcessRunner` is a real subprocess, env-detected

For the laptop binding, `LocalProcessRunner.submit()` actually shells out to
`giant.quick_refine`. We probe for it (`shutil.which("giant.quick_refine")`) at
launch; if absent, the **dispatch feature is disabled in the UI** with a clear
"needs CCP4 on PATH" message. The inspect + build-ligand loop still works
without CCP4 — only dispatch is gated. Honest end-to-end where CCP4 exists,
degrades cleanly where it does not.

`submit` must return immediately with a handle and the client polls `status` —
never block on the subprocess. (This is also what the cross-arch compose runner
forces; see §3.3 — a synchronous "run and wait" runner would be exposed as wrong
the moment it ran emulated on a Mac.)

## 3. Two deployment bindings — the proof, not just a target

Same `inspect_api/`. The bindings differ only at the two seams + packaging:

| Concern | Electron (laptop) | docker-compose (server) | What the diff proves |
|---|---|---|---|
| **DataStore** | `LocalFileStore` on host FS | `LocalFileStore` on a mounted **volume** | bytes-location is config, not code |
| **JobRunner** | `LocalProcessRunner` → host CCP4 | `SharedVolumeRunner` → amd64 CCP4 sidecar | compute-launch is the *most* backend-specific seam — and it's pluggable |
| **DB** | SQLite file | SQLite on a volume (or Postgres) | transactional store is swappable |
| **Backend process** | PyInstaller binary spawned by Electron main | `gunicorn`/`runserver` in a container | packaging ≠ contract |

The **DataStore proof is nearly free**: `LocalFileStore` pointed at a
bind-mounted volume *is* the proof — same class, different root.

### 3.1 Electron binding — packaging decision: bundle a frozen Python runtime

**Decision: PyInstaller-freeze the backend; Electron main spawns the binary.**
Reasoning:

- The point of the handover is "a colleague runs it without ceremony."
  Requiring a compatible *system* Python defeats that for the read loop — which
  must work for everyone, CCP4 or not.
- Our backend is unusually freeze-friendly because the native/scientific weight
  is **already pushed out of the backend process**: CCP4 is an external
  subprocess, Moorhen is WASM in the client. So PyInstaller freezes just
  Django + DRF + drf-spectacular + sqlite — small, pure-Python, none of the
  usual numpy/native-wheel pain.
- It keeps the story clean: **Electron is the laptop binding of the contract.**
  A self-contained binary *is* that story; a "go install Python first" readme is
  a leak.
- Cost is honest: ~30–50 MB frozen Python on top of an Electron app already
  large because of Moorhen's WASM/monomer-library payload. Moorhen dominates;
  the frozen Python is in the noise.

CI (GitHub Actions, macOS runner): build the PyInstaller binary → electron-builder
packages the installer → downloadable artifact.

### 3.2 docker-compose binding — sidecar runner from the start

Two services sharing a volume:

```yaml
services:
  web:                          # the contract-serving backend
    build: ./backend            # NO platform pin -> host-native
                                #   (arm64 on a Mac, amd64 on Linux/CI)
    volumes: [data:/data, jobs:/jobs]
  runner:                       # the compute backend
    platform: linux/amd64       # CCP4 reality; emulated on ARM, native on Intel/CI
    image: <ccp4-runner>
    volumes: [data:/data, jobs:/jobs]
volumes: { data: {}, jobs: {} }
```

`web` writes a JobSpec + input artifacts to the `jobs` volume and returns a
handle immediately; `runner` watches the volume, execs `giant.quick_refine`,
writes output bytes + a status file back; `web` polls and, on success, registers
the output as an `Artifact` and repoints `Dataset.current_model`. This
`SharedVolumeRunner` is a **second, distinct `JobRunner` implementation** living
side-by-side with the laptop's `LocalProcessRunner` — which is the seam proof.

**Why the sidecar is mandatory, not optional — the cross-arch argument.** Linux
CCP4 is strictly amd64. On an Apple-Silicon dev machine (several of us, incl.
the maintainer), an amd64 image runs under QEMU emulation. If CCP4 were baked
into the `web` image, the **entire backend** would be forced to amd64 and run
emulated — every API call and SQLite hit paying the QEMU tax for nothing.
Splitting the runner out means **only the compute that genuinely needs amd64
pays the emulation cost; the contract-serving backend stays native.** The
architecture seam and the architecture *constraint* point the same way.

### 3.3 Cross-architecture rules (enabled from the start)

- **Pin platform per service, never globally.** `web` host-native; `runner`
  pinned `linux/amd64`.
- **Decouple dispatch from execution speed.** The emulated runner is *slow*, so
  `submit` must return a handle and the client polls — the `JobRunner` protocol
  already mandates this, so the constraint *validates* the seam's shape.
- **Dispatch transport is a shared-volume job queue**, chosen precisely because
  it is architecture-agnostic: a JobSpec file appearing on a volume does not care
  that the writer is arm64 and the reader is amd64-emulated. Emulation is
  isolated to the single `exec` inside the runner. (Deliberately *not*
  docker-socket/exec: that couples `web` to the docker engine, needs socket
  access, and generalises poorly to SLURM/cloud.)
- **binfmt/QEMU required for ARM hosts.** Docker Desktop ships it; for a plain
  Linux engine document `docker run --privileged tonistiigi/binfmt`.
- **Perf caveat stated honestly.** Emulated CCP4 is for *proving the loop*, not
  throughput. A real lab deploy runs the runner on amd64 hardware (or a
  SLURM/cloud runner) — the *next* `JobRunner` impl, same seam. Note the nice
  inversion: the Mac-dev experience is the slow path, the server/CI experience
  is the fast path — different perf profiles, one contract.
- **CI builds amd64 natively** (GitHub Linux runners are amd64), so the runner
  image is fast in CI and only emulated on ARM laptops.

This shared-volume `SharedVolumeRunner` is also the cleanest preview of how
CCP4Cloud / CCP4i2 would plug in: "dispatch a JobSpec to an executor that lives
elsewhere and poll a handle" is the same shape whether the executor is a sibling
container, a SLURM queue, or a cloud API. Swapping the transport later (docker
exec, Redis/RQ, a cloud API) moves neither the `JobRunner` interface nor the
artifact-landing contract.

## 4. Implementation order (proposed, post-design)

1. **Schema** — add `Artifact.origin/parent/produced_by/created_at`,
   `Event.current_model`, `Dataset.current_model`, `inputs_changed` flags, and
   the `Job` model + migration. (No behaviour yet — just the durable shape.)
2. **Re-ingest policy** — make both ingest readers honour §1.3 (additive,
   import-scoped, flag divergence) instead of clobbering.
3. **`JobRunner` real** — `LocalProcessRunner.submit/status/cancel` over a real
   subprocess; env-detect `giant.quick_refine`; the submit→poll→land-artifact
   loop (§2.1) wired into a `Job` viewset + endpoints.
4. **#4 ligand build** — port the Coot recipe; register `origin=built` artifact;
   auto-PATCH decision=hit (§2.2).
5. **Electron binding** — main process spawns the PyInstaller backend; wire
   `LocalFileStore` + `LocalProcessRunner`; CI installer (§3.1).
6. **docker-compose binding** — `web` + amd64 `runner` sidecar + shared-volume
   `SharedVolumeRunner`; cross-arch rules (§3.2–3.3).

Items 1–2 are the #2 foundation and should land before 3–4 (a job/build that
produced artifacts *before* the lineage model existed would recreate the drift
bug). 5 and 6 are independent bindings of whatever 1–4 produce.
